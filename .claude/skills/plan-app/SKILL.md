このプロジェクトに対して、指定された変更の計画書を作成してください。

## 入力

$ARGUMENTS

## 実装開始前の確認

変更内容に曖昧な点・不明な点がある場合は、**計画書を作成する前に必ず質問すること。**
以下のような場合は質問が必要：

- 要件の解釈が複数通りある
- 変更範囲が不明確（どこまで変更すべきか判断できない）
- 既存の設計と矛盾する指示がある
- 実装に必要な情報（仕様・データ形式・API 等）が不足している

複数の疑問点は1つのメッセージにまとめて質問すること。

## 変更計画書の作成

**必ず変更計画書を `.claude/docs/<YYYY-MM-DD>-<変更内容の要約>.md` に出力すること。**

計画書には以下を含めること：

```markdown
# 変更計画書: <タイトル>

## 概要

変更の目的と背景を簡潔に説明する。

## 変更対象ファイル

変更・追加・削除するファイルの一覧と、各ファイルへの変更内容を記述する。

| ファイル         | 変更種別 | 変更内容 |
| ---------------- | -------- | -------- |
| src/extension.ts | 変更     | ...      |
| package.json     | 変更     | ...      |

## 実装方針

設計上の判断や方針を記述する。新機能を追加する場合は、既存の `src/` 構成（機能ごとに独立したファイル、`extension.ts` でアクティベーション・登録）に従うかどうかを明記すること。また `package.json` の `contributes.configuration` への追加が必要かどうかも記述すること。

## 副作用・注意事項

既存の振る舞いが変わる箇所や、影響範囲を記述する。

## Git運用

使用するブランチ名と、コミットの分割方針を記述する。コミットメッセージはコンベンショナルコミット形式（`feat:` / `fix:` / `chore:` など）を使用すること。
```

計画書を作成した後、ユーザーに内容を確認し、**承認が得られるまで実装は行わないこと。**
