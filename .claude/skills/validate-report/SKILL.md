---
name: validate-report
description: plan-appが出力した.claude/docs/内の変更計画書を検証し、精度・実現可能性・プロジェクト整合性を確認する
---

`.claude/docs/` に出力された変更計画書の内容が適切かどうかを検証してください。

## 検証対象

$ARGUMENTS で指定されたファイルを対象とする。

## 検証観点

**推測・記憶・想像による判断は禁止。** 必ずファイルを開いて該当箇所を目視確認すること。

### 1. 事実確認（Fact-Check）

- 「変更対象ファイル」テーブルに記載されたファイルパスが実際に存在するか（新規追加ファイルを除く）
- 変更内容の説明が実際のコードと矛盾していないか
- 「副作用・注意事項」に記載された影響範囲が実際のコードに照らして正確か

### 2. 網羅性確認（Completeness-Check）

- 計画書の構成（概要・変更対象ファイル・実装方針・副作用・Git運用）がすべて揃っているか
- 変更に連動して修正が必要なファイルが「変更対象ファイル」から漏れていないか
  - 例: 新機能追加時に `src/extension.ts`（登録処理）や `src/types.ts`（型定義）の修正が必要なのに記載がない
- 「副作用・注意事項」が空欄または「なし」の場合、本当に副作用がないかコードを確認する

### 3. 実装方針確認（Design-Check）

- 新機能を追加する場合、既存の `src/` 構成（機能ごとに独立した `.ts` ファイル、`src/extension.ts` でアクティベーション・登録、`src/types.ts` に型定義）に従っているか
- 既存コードの責務分担（`src/types.ts` は型定義のみ、各機能ファイルは独立したモジュール）を逸脱していないか
- 新しい設定項目を追加する場合、`package.json` の `contributes.configuration` への追加が計画に含まれているか
- 不要な依存関係の追加や過剰なリファクタリングが計画に含まれていないか

### 4. プロジェクト整合性確認（Consistency-Check）

- 提案内容が既存の設計思想を壊すものになっていないか
- 提案内容が動作中の機能を破壊・劣化させるものになっていないか（リグレッションリスクを含む）
- `change-app` スキルの Git 運用ルール（ブランチ命名規則・コンベンショナルコミット・lint 実行・main マージ手順）と一致しているか

### 5. バグリスク確認（Bug-Risk-Check）

計画書が提案する変更を適用した場合、新たなバグが発生しないかを以下の観点で確認してください。

- **副作用の有無**: 変更対象以外のコード（呼び出し元・依存モジュール・共有状態）に意図しない影響を与えないか
- **エッジケースの未考慮**: 空値・nil・ゼロ値・エラー時などのケースが考慮されているか
- **型・インターフェース整合性**: 変更後も呼び出し元が期待する型・シグネチャと一致しているか
- **並行安全性**: 変更によって競合状態やデッドロックが生じる可能性はないか

## 出力形式

標準出力（チャット上）に以下の構成で出力してください（ファイル出力は不要）。

```
## 検証結果: <対象ファイル名>

### 事実確認
- [OK/NG] <確認項目>: <根拠（実際に確認したファイルパス）>
...

### 網羅性
- 漏れあり: <漏れているファイルや項目とその根拠>（なければ「漏れなし」）

### 実装方針
- [OK/NG] <確認項目>: <コメント>
...

### プロジェクト整合性
- [OK/NG] <確認項目>: <コメント>
...

### バグリスク
- [OK/NG] <確認項目>: <根拠または懸念点>
...

### 総合評価
**信頼度**: 高 / 中 / 低
**主な問題**: <NGがあった場合の要約>（なければ「なし」）
**推奨アクション**: <そのまま change-app に渡してよい / 計画書の修正が必要 / plan-app を再実行すべき>
```
