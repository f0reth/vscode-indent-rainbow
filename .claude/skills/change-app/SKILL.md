---
name: change-app
description: 計画書に基づきアプリの変更を実施
---

このプロジェクトに対して、指定された計画書または内容に基づき変更を実施してください。

## 入力

$ARGUMENTS

入力が計画書のファイルパスの場合はそのファイルを読み込み、テキストの場合はそのまま変更内容として扱うこと。

## 目的

- 指定された要件に従ってアプリを変更する
- 既存の設計思想を維持する
- 不要な複雑さを排除し、可読性と保守性を向上させる

## 実装ルール

- 変更は最小単位で行う（過剰なリファクタリングは禁止）
- 同時に改善できる軽微なリファクタリングは許可（例：命名修正、未使用インポートの削除、typo修正）
- 命名は既存コードの規則に従い一貫性を維持する
- 不要な依存関係を追加しない
- 副作用のある変更（既存の振る舞いが変わる箇所）はコミット本文に `BREAKING:` または `NOTE:` で始まる行を追加して明示する

## Git運用ルール

以下を必ず守ること：

1. mainブランチから新しいブランチを作成する

- 命名規則: `feat/<変更内容の要約>` または `fix/<変更内容の要約>`
  - 例: `feat/add-user-validation`, `fix/user-validation`
- 計画書にブランチ名が記載されている場合はそれに従う

2. 論理単位でコミットを分割する（例：型定義の追加・機能実装・設定追加は別コミット）
   - 計画書にコミット分割方針が記載されている場合はそれに従う
3. コミットメッセージはコンベンショナルコミット形式を使用する（`feat:` / `fix:` / `chore:` など）
4. コミット前に必ず以下を実行し、エラーがないことを確認する（必須）

```bash
pnpm run fmt && pnpm run lint
```

- `pnpm run lint` は oxlint による型チェック付き静的解析（`oxlint --type-aware --type-check`）
- エラーが出た場合は修正してから再実行し、必ずクリーンな状態でコミットすること
- `pnpm run fmt` で変更されたファイルはコミット対象に含める
- エラーが出た場合は修正して再実行し、エラーが0件になったらコミットしてステップ5へ進む
- **警告**: エラーが残ったままコミットしてはいけない

5. すべての変更が完了したら、mainブランチにマージしてブランチを削除する

```bash
git switch main
git merge <ブランチ名>
git branch -d <ブランチ名>
```

6. マージ後の最終確認（必須）

すべての変更・マージが終わったら、必ず以下を実行してエラーがないことを確認する：

```bash
pnpm run fmt && pnpm run lint
```

- エラーが出た場合は修正し、mainブランチ上でコミットしてからこのステップを繰り返す
- エラーが0件になったら作業完了とする
